# トラブルシューティング

運用中によくある症状と、その原因・対処をまとめました。困ったときは、まず**該当する症状**を探し、上から順に試してください。

> まず落ち着いて: 送信に関する異常を感じたら、**先に送信を止めて**から原因を調べると安全です。
> ```bash
> mkdir -p data/state && touch data/state/STOP   # kill switch（即時に送信停止）
> ```
> 解決後、`rm data/state/STOP` で再開できます。`data/state` の部分は `.env` の `BIZSCOUT_KILL_SWITCH` の設定に合わせてください（既定 `data/state/STOP`）。

---

## まず確認する共通の3点

どの症状でも、最初に次の3つを確認すると原因の切り分けが早くなります。

1. **ログを見る** … `logs/bizscout.log`（cron 利用時は `logs/resend.log` など）。詳細を出すには各コマンドに `--verbose` を付けます。
   ```bash
   bizscout --verbose run --source bizreach --search-url "..."
   tail -f logs/bizscout.log
   ```
2. **状況を見る** … `bizscout report` で件数と「要確認」候補者を確認します。
3. **起動前チェック** … `bizscout doctor` で設定の不足や不整合を確認します。

## 症状1: ログインに失敗する

### よくあるメッセージ・状況
- `BIZREACH_EMAIL / BIZREACH_PASSWORD が未設定です。ログインできません。`
- ログイン後に「ログイン成功の確認要素が見つかりません」という警告がログに出る。
- ログイン画面で止まってしまう。

### 原因と対処

1. **認証情報が未設定／誤り**
   `.env` の `BIZREACH_EMAIL` と `BIZREACH_PASSWORD` を確認してください。前後の空白や全角文字に注意します。
2. **2段階認証が有効**
   メール・パスワードだけでは自動ログインできません。一度だけ手動でログインしてセッションを保存します。
   ```bash
   bizscout run --source bizreach --search-url "..." --no-headless
   ```
   表示されたブラウザで手動ログイン（認証コード入力含む）すると、`.env` の `BIZREACH_STORAGE_STATE`（既定 `data/sessions/bizreach_state.json`）にセッションが保存され、以降は再利用されます。詳しくは **docs/セレクタ設定ガイド.md** の「6」を参照。
3. **ログイン成功の目印がずれている**
   ログインできているのに「確認要素が見つかりません」と出る場合、`logged_in_marker` が実画面と合っていません。**docs/セレクタ設定ガイド.md** の手順で実画面に合わせて `config/bizreach_selectors.yaml` を更新してください。
4. **保存セッションの期限切れ**
   以前は動いていたのに急にログインを求められる場合、保存セッションが期限切れの可能性があります。`--no-headless` でもう一度手動ログインして更新してください。古いセッションを消したい場合は `data/sessions/bizreach_state.json` を削除してから再ログインします。
5. **ログインURLが違う**
   `login_url` がアカウント種別により異なることがあります。実際のログインURLを `config/bizreach_selectors.yaml` の `login_url` に設定してください。

## 症状2: セレクタが一致しない（ボタン・入力欄が見つからない）

### よくあるメッセージ・状況
- ログに `Timeout` や「要素が見つからない」系のエラーが出る。
- 件名・本文は入るのに送信ボタンで止まる、など特定の操作だけ失敗する。

### 原因と対処

ビズリーチの画面更新などで、目印（セレクタ）が実画面と合わなくなった状態です。

1. **どの操作で止まったかをログで特定**
   `--verbose` を付けて実行し、どの段階（ログイン／検索／プロフィール／件名／本文／送信）で止まったかを確認します。
2. **該当項目を実画面に合わせて更新**
   止まった操作に対応する項目を、**docs/セレクタ設定ガイド.md** の手順（開発者ツールで検証 → YAML に記入）で更新します。例:
   - スカウト作成が開かない → `scout_button`
   - 件名／本文が入らない → `scout_subject` / `scout_body`
   - 送信できない → `scout_send` / `scout_confirm`
   - 検索結果から進めない → `result_link` / `next_page`
3. **候補を複数並べる**
   `,`（カンマ）で複数候補を並べると当たりやすくなります（例: `"[data-testid='scout-send'], button:has-text('送信')"`）。
4. **画面の見え方を確認したいとき**
   `--no-headless` を付けて実行すると、ブラウザ画面が見える状態になり、どこで止まっているか目視できます。
5. **更新後は必ずドライラン**
   セレクタを直したら、`BIZSCOUT_DRY_RUN=true` のまま少件数で動作確認してから本番に戻します。

> `bizscout doctor` でも設定の不足を確認できます。直したら再度 `doctor` を実行してください。

## 症状3: 送信されない（文面は作られるのに送られない）

文面の入力までは進むのに送信されない場合、多くは**安全装置が働いている**だけで、故障ではありません。次の順で確認します。

1. **ドライランが有効になっていないか**
   `.env` の `BIZSCOUT_DRY_RUN` が `true` だと、文面入力までで止まり**送信ボタンは押されません**。実行時に次の案内が出ていないか確認してください。
   ```
   ※ DRY_RUN 有効: 文面は入力しますが実送信は行いません（.env の BIZSCOUT_DRY_RUN）。
   ```
   本番送信したい場合は `BIZSCOUT_DRY_RUN=false` にします。
2. **kill switch が残っていないか**
   `BIZSCOUT_KILL_SWITCH` のパス（既定 `data/state/STOP`）にファイルが残っていると、送信は全停止します。不要なら削除します。
   ```bash
   rm data/state/STOP
   ```
3. **送信上限に達していないか**
   `BIZSCOUT_MAX_SENDS_PER_RUN` の件数に達すると、残りは次回に持ち越されます。ログに「送信上限に達した」旨が出ていないか確認し、必要なら上限を見直します。
4. **送信元（ソース）が bizreach か**
   自動送信は `--source bizreach` のみ対応です。`csv` / `text` は**生成・保存のみ**で、送信は行いません（実行時に案内が出ます）。送信したい場合は bizreach ソースを使ってください。
5. **対象条件で除外されていないか**
   対象条件（年齢・在籍年数・性別・学歴など）を満たさない、または不明の候補者は自動送信から除外され、「要確認」として記録されます。`bizscout report` の「要確認」一覧で確認できます。
6. **プロフィールURLが取れていないか**
   送信先のプロフィールURLが取得できていない候補者はスキップされます（とくに再送時）。ログに `no profile_url` 等が出ていないか確認します。
7. **送信完了マーカーが確認できない**
   ログに「送信完了マーカーが確認できませんでした」と出る場合、送信自体は試みられたが完了の判定要素が見つからなかった状態です。`scout_sent_marker` を実画面に合わせて更新してください（**docs/セレクタ設定ガイド.md**）。

## 症状4: レート制限・bot検知が疑われる

### よくあるメッセージ・状況
- 途中から急に操作が失敗する、画面に確認（CAPTCHA）や警告が出る、一時的にアクセスできなくなる。

### 原因と対処

短時間に多く送りすぎると、サービス側の制限や bot 検知が働くことがあります。

1. **送信を一旦止める**
   まず kill switch で止めます。
   ```bash
   mkdir -p data/state && touch data/state/STOP
   ```
2. **送信間隔を広げる**
   `.env` の `BIZSCOUT_SEND_DELAY_MIN` / `BIZSCOUT_SEND_DELAY_MAX` を大きくし、送信ごとの待機を長くします（より人間らしい間隔になります）。
3. **1回あたりの件数を減らす**
   `BIZSCOUT_MAX_SENDS_PER_RUN` を小さくし、1日の総量も控えめにします。
4. **時間を空ける**
   一時的な制限であれば、しばらく時間を空けてから少件数で再開し、様子を見ます。
5. **運用ポリシーを再確認**
   ビズリーチの利用規約・自動化に関するポリシーを確認し、無理のない範囲で運用してください。過度な送信はアカウント制限の対象になり得ます。

> 再開するときは kill switch を外し（`rm data/state/STOP`）、まず少件数で動作を確認してから通常運用に戻すと安全です。

## 症状5: 再送が走らない

### よくあるメッセージ・状況
- `bizscout run-resends` を実行しても「対象(期限到来): 0」と出る／何も送られない。

### 原因と対処

1. **まだ期限が来ていない**
   再送は初回送信から `BIZSCOUT_RESEND_AFTER_DAYS` 日（既定5日）が経過した候補者だけが対象です。期限前は「対象 0」になります。日付を確認してください。
2. **そもそも初回が送信済みか**
   初回が**実際に送信済み**になっていないと、再送はスケジュールされません。ドライラン（`DRY_RUN=true`）で動かした分は実送信されていないため、再送対象にはなりません。`bizscout report` で「送信済み」件数を確認してください。
3. **送信オフで実行していないか**
   `bizscout run-resends --no-send` だと送信しません。送信したい場合はオプションなしで実行します。
4. **ドライラン・kill switch・送信上限**
   再送も `BIZSCOUT_DRY_RUN` / `BIZSCOUT_KILL_SWITCH` / `BIZSCOUT_MAX_SENDS_PER_RUN` に従います。送信されないときは症状3と同じ点を確認してください。
5. **プロフィールURLが無い**
   再送先のプロフィールURLが無い候補者はスキップされます（ログに `no profile_url`）。
6. **定期実行が動いていない**
   cron / systemd / `bizscout serve` などの定期実行が止まっていると、再送も実行されません。設定と稼働状況を確認してください（定期実行の構成は `deploy/` を参照）。

## それでも解決しないとき

- `bizscout --verbose ...` で詳細ログを取り、`logs/` の内容を確認してください。
- セレクタ起因が疑われる場合は **docs/セレクタ設定ガイド.md** の手順で実画面に合わせて更新してください。
- 全体の運用手順は **docs/運用手順.md** にまとまっています。
