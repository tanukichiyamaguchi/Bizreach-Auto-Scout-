# 運用手順（完全自動運用 / 使い方B）

この手順書は、**採用担当者の方がご自身で「取り込み → 文面生成 → ビズリーチへの自動送信 → 再送」までを自動運用する**ための、最初から最後までの手順をまとめたものです。専門的な知識がなくても、上から順にそのまま進めれば運用を始められるよう、できるだけ具体的に書いています。

> ⚠️ いちばん大切なこと
> スカウト送信は**一度送ると取り消せません**。本番送信に切り替える前に、必ず本書の「6. 初回ドライラン」までを実施し、生成された文面と動作を目視で確認してください。`BIZSCOUT_DRY_RUN=true`（既定値）の間は、文面の入力までは行いますが**送信ボタンは押されません**。安心して練習できます。

---

## 0. この手順の全体像

実際の運用は、おおまかに次の流れで進みます。

1. 実行用のパソコン（またはサーバー）を用意する
2. プログラムをインストールする
3. `.env` ファイルに設定を書く（まずは送信しない安全モード）
4. ビズリーチ画面の「セレクタ」を設定する（別ガイド参照）
5. `bizscout doctor` で起動前チェックをする
6. 安全モードのまま試運転（ドライラン）して、文面を確認する
7. 問題なければ本番送信に切り替える
8. 定期実行のしくみを整える（常駐 / cron / systemd / Docker）
9. 日々の監視と、止めたいときの操作を覚える

各ステップを以降で順に説明します。

---

## 1. 必要なもの（全体像）

運用を始める前に、以下の4つを用意してください。

| 区分 | 内容 | 備考 |
|------|------|------|
| 実行マシン | プログラムを動かすパソコンまたはサーバー | macOS / Linux / Windows。常時稼働なら社内サーバーやクラウドが向きます。インターネット接続が必要です。 |
| Claude API キー | 文面生成に使う Anthropic（Claude）の API キー | `ANTHROPIC_API_KEY`。Anthropic のアカウントで発行します。 |
| ビズリーチ認証情報 | 採用企業アカウントのメールアドレスとパスワード | `BIZREACH_EMAIL` / `BIZREACH_PASSWORD`。自動ログインに使います。2段階認証がある場合は本書「7-4」と別ガイドの手順を参照。 |
| 実セレクタ | ビズリーチ画面の「どのボタンか」を特定する設定 | `config/bizreach_selectors.yaml`。詳細は **docs/セレクタ設定ガイド.md** を参照。 |

> ビズリーチのブラウザ自動操作は、**利用規約・自動化に関するポリシーを確認のうえ、ご自身のアカウント・責任で**運用してください。過度な送信はアカウント制限の対象になり得ます。送信間隔や1回あたりの上限は本書「7」で必ず設定してください。

## 2. インストール

実行マシンのターミナル（コマンド入力画面）で、リポジトリのフォルダに移動してから、以下を順に実行します。

```bash
# 1) 仮想環境（このプロジェクト専用の Python 環境）を作って有効化
python3 -m venv .venv
source .venv/bin/activate          # Windows の場合: .venv\Scripts\activate

# 2) 本体と依存ライブラリをインストール
pip install -e .

# 3) ブラウザ自動操作用の Chromium をインストール
playwright install chromium
```

- `pip install -e .` … このプロジェクトをインストールし、`bizscout` コマンドが使えるようになります。
- `playwright install chromium` … ビズリーチを自動操作するためのブラウザ本体を入れます。自動送信を使うなら必須です。

正しく入ったかは、次で確認できます。

```bash
bizscout --help
```

サブコマンドの一覧が表示されれば成功です。

## 3. `.env` の設定

設定はすべて、リポジトリ直下の `.env` というファイルに書きます。まずは雛形をコピーします。

```bash
cp .env.example .env
```

`.env` をテキストエディタで開き、以下を埋めてください。**最初は必ず `BIZSCOUT_DRY_RUN=true`（送信しない安全モード）のままにします。**

```ini
# --- Claude (Anthropic) API ---
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxxxxxx   # ← あなたの API キー
BIZSCOUT_MODEL=claude-opus-4-8                  # 生成モデル（品質重視は opus、コスト重視は sonnet 系）

# --- ビズリーチ ログイン情報 ---
BIZREACH_EMAIL=your-recruiter-account@example.com
BIZREACH_PASSWORD=your-password
BIZREACH_STORAGE_STATE=data/sessions/bizreach_state.json   # ログインセッションの保存先

# --- 送信制御 ---
BIZSCOUT_DRY_RUN=true            # true=送信しない（最初は必ずこれ）
BIZSCOUT_MAX_SENDS_PER_RUN=20    # 1回の実行で送る最大件数（暴走防止）
BIZSCOUT_SEND_DELAY_MIN=20       # 送信と送信の間の待機（秒）の下限
BIZSCOUT_SEND_DELAY_MAX=60       # 同上の上限
BIZSCOUT_KILL_SWITCH=data/state/STOP   # このファイルがある間は送信を全停止

# --- 再送 ---
BIZSCOUT_RESEND_AFTER_DAYS=5     # 初回送信から何日後に再送するか

# --- パス類（通常は変更不要）---
BIZSCOUT_HOME=.
BIZSCOUT_DB_PATH=data/bizscout.db
```

各変数の意味は次のとおりです。

| 変数 | 意味 |
|------|------|
| `ANTHROPIC_API_KEY` | Claude（Anthropic）の API キー。文面生成に使います。 |
| `BIZSCOUT_MODEL` | 文面生成に使うモデル名。既定は `claude-opus-4-8`。 |
| `BIZREACH_EMAIL` / `BIZREACH_PASSWORD` | ビズリーチ採用企業アカウントのログイン情報。 |
| `BIZREACH_STORAGE_STATE` | ログイン後のセッション保存先。これがあると毎回ログインせずに済みます。 |
| `BIZSCOUT_DRY_RUN` | `true` の間は実送信せず、文面の入力までで止めます（最重要の安全装置）。 |
| `BIZSCOUT_MAX_SENDS_PER_RUN` | 1回の実行で送る上限件数。暴走を防ぎます。 |
| `BIZSCOUT_SEND_DELAY_MIN` / `BIZSCOUT_SEND_DELAY_MAX` | 送信ごとの待機秒数の下限・上限。ランダムな間隔でレート制限・bot検知を避けます。 |
| `BIZSCOUT_KILL_SWITCH` | このパスにファイルを作ると、即時に送信を全停止します（緊急停止）。 |
| `BIZSCOUT_RESEND_AFTER_DAYS` | 初回送信から再送までの日数（既定5）。 |
| `BIZSCOUT_HOME` / `BIZSCOUT_DB_PATH` | 作業ディレクトリとデータベースの場所。通常は変更不要です。 |

> `.env` には API キーやパスワードが入ります。**他人と共有せず、リポジトリにもコミットしない**でください（`.env` は最初から除外設定済みです）。

## 4. セレクタ設定（ビズリーチ画面のボタン特定）

プログラムは、ビズリーチ画面の「ログインボタン」「スカウト送信ボタン」などを **セレクタ** という目印で探します。ビズリーチの画面は変わることがあるため、実際の画面に合わせた設定が必要です。

雛形をコピーします。

```bash
cp config/bizreach_selectors.example.yaml config/bizreach_selectors.yaml
```

`config/bizreach_selectors.yaml` を開き、実際の画面に合わせて値を記入します。**空欄のままにすると、プログラム内蔵の初期値（`src/bizreach_scout/bizreach/selectors.py` の値）が使われます。** 初期値はあくまで「想定値」なので、実運用前に必ず実際の値へ更新してください。

具体的な見つけ方・書き方は **docs/セレクタ設定ガイド.md** に詳しく書いています。先にそちらを完了させてから次へ進んでください。

---

## 5. `bizscout doctor`（起動前チェック）

設定が一通り終わったら、本番の前に起動前チェックを実行します。

```bash
bizscout doctor
```

`doctor` は、API キーやビズリーチ認証情報、セレクタ設定、必要なファイルやフォルダがそろっているかなどを確認し、不足があれば指摘します。指摘が出たら、その内容に従って `.env` やセレクタ設定を直してから、再度 `doctor` を実行してください。すべて問題なしになってから次へ進みます。

## 6. 初回ドライラン（安全モードでの試運転）

`.env` の `BIZSCOUT_DRY_RUN=true` のまま、実際に取り込み〜文面生成までを動かして確認します。**この段階では送信ボタンは押されません。**

```bash
bizscout run --source bizreach \
  --search-url "https://cr-support.jp/search?saved=..." \
  --max 5
```

- `--source bizreach` … ビズリーチ画面を自動操作して候補者を取り込みます。
- `--search-url` … ビズリーチで作った保存検索の結果ページのURLを指定します。
- `--max 5` … まずは少なめ（例: 5件）で試すと安全です。

`DRY_RUN=true` のときは、画面に次の案内が出ます。

```
※ DRY_RUN 有効: 文面は入力しますが実送信は行いません（.env の BIZSCOUT_DRY_RUN）。
```

実行が終わると、処理件数のサマリが表示されます。続けて、生成された文面や送信状況を確認します。

```bash
# 特定の候補者（会員番号）の文面を確認（初回用・再送用の両方）
bizscout preview BU3765516

# 全体の送信状況サマリと「要確認」候補者の一覧を確認
bizscout report
```

- `bizscout preview <会員番号>` … 保存済みのスカウト文面（件名・本文）を、初回送信用と再送用の順で表示します。
- `bizscout report` … `生成済み / 送信済み` などの状況件数と、対象条件を満たさず自動送信から除外された「要確認」候補者を表示します。

> 文面そのものをファイルでじっくり見たい場合は、`bizscout generate --source csv --input ... --save` を使うと `data/exports/<会員番号>.md` にも書き出せます。送信は一切行いません。

ここで**件名・本文の内容、宛先の妥当性、要確認候補者の扱い**をしっかり確認してください。納得できる品質になるまで、設定や検索条件を調整しながら、この安全モードでの試運転を繰り返します。

## 7. 本番送信への切り替え

ドライランで品質と動作に納得できたら、本番送信に切り替えます。

### 7-1. 送信上限・間隔を設定する

`.env` で、暴走防止とレート制限対策の値を確認・調整します。

```ini
BIZSCOUT_MAX_SENDS_PER_RUN=20   # 1回の実行で送る上限（最初は小さめが安全）
BIZSCOUT_SEND_DELAY_MIN=20      # 送信間隔の下限（秒）
BIZSCOUT_SEND_DELAY_MAX=60      # 送信間隔の上限（秒）
```

最初は上限を小さめ（例: 5〜10件）にし、様子を見ながら少しずつ増やすことをおすすめします。

### 7-2. ドライランを解除する

```ini
BIZSCOUT_DRY_RUN=false
```

`false` にすると、次回の実行から**実際に送信ボタンが押されます。** ここからは取り消せない操作になります。慎重に進めてください。

### 7-3. 本番実行

```bash
bizscout run --source bizreach --search-url "https://cr-support.jp/search?saved=..." --max 10
```

実行後は必ず `bizscout report` で送信件数を確認します。

### 7-4. 緊急停止（kill switch）の使い方

送信中・送信前に**今すぐ止めたい**ときは、`.env` の `BIZSCOUT_KILL_SWITCH` で指定したパス（既定 `data/state/STOP`）に**ファイルを1つ作るだけ**で、それ以降の送信が止まります。

```bash
# 緊急停止：このファイルがある間は一切送信しない
mkdir -p data/state && touch data/state/STOP
```

再開したくなったら、そのファイルを削除します。

```bash
rm data/state/STOP
```

> kill switch は「送信」だけを止めます。すでに送信済みのものは取り消せません。`.env` の編集（`DRY_RUN=true` に戻す）よりも即効性があるので、緊急時はまずこの方法で止めてください。

### 7-5. 再送

初回送信から `BIZSCOUT_RESEND_AFTER_DAYS` 日（既定5日）が経った候補者へ、再送を送ります。

```bash
bizscout run-resends
```

期限が来ていない候補者は対象外です。これも `BIZSCOUT_DRY_RUN` と送信上限・間隔の設定に従います。

## 8. 定期実行のしくみ

完全自動運用では、上記コマンドを毎日決まった時刻に自動実行します。環境に合わせて、次のいずれかを選びます。

### 選択肢A: `bizscout serve`（常駐）

```bash
bizscout serve
```

`serve` はプロセスを常駐させ、初回送信と再送のスケジュールを内部で管理して定期実行します。実行マシンを常時起動しておける場合に向いています。

### 選択肢B: cron（Linux / macOS）

OS の cron に登録して、決まった時刻に実行します。

```cron
# 毎朝9時に再送を実行し、ログを追記
0 9 * * *  cd /path/to/Bizreach-Auto-Scout- && .venv/bin/bizscout run-resends >> logs/resend.log 2>&1
```

初回送信を毎朝動かす場合も、同様に `bizscout run --source bizreach --search-url "..."` を1行追加します。

### 選択肢C: systemd（Linux サーバー常駐）

サーバーで安定して常駐させたい場合は systemd を使います。サービス定義の雛形は `deploy/systemd/` に用意されています（インストール手順を含む）。

### 選択肢D: Docker

コンテナで動かす場合は `deploy/` 配下の構成を参照してください。

> 定期実行の具体的な設定ファイルや手順は、リポジトリの **`deploy/`** ディレクトリにまとめられています。導入時はそちらをご確認ください。

## 9. 監視と、止めたいときの操作

### 9-1. ログを見る

実行ログは `logs/` フォルダに記録されます（例: `logs/bizscout.log`、cron 利用時は `logs/resend.log` など）。送信の成否やエラーはここで確認できます。

```bash
# 直近のログをリアルタイム表示（Ctrl+C で終了）
tail -f logs/bizscout.log
```

> ログを詳細に出したいときは、各コマンドに `--verbose` を付けて実行します（例: `bizscout --verbose run ...`）。

### 9-2. 送信状況を見る

```bash
bizscout report
```

`生成済み / 送信済み / 失敗` などの件数と、自動送信から除外された「要確認」候補者を確認できます。日次運用では、毎朝このコマンドで前日分を点検すると安心です。

### 9-3. すぐ止めたいとき

- **即時に止める**: kill switch ファイルを作る（本書 7-4）。
  ```bash
  mkdir -p data/state && touch data/state/STOP
  ```
- **しばらく送信を止める**: `.env` で `BIZSCOUT_DRY_RUN=true` に戻す。次回実行から送信されません。
- **常駐を止める**: `bizscout serve` を実行している場合は、そのプロセスを停止します（systemd 運用なら該当サービスを停止）。

異常を感じたら、まず kill switch で送信を止め、`bizscout report` と `logs/` で状況を確認してから原因を切り分けてください。困ったときの症状別の対処は **docs/トラブルシューティング.md** にまとめています。

---

## 困ったときは

- ビズリーチ画面のボタンが見つからない・ログインできない → **docs/セレクタ設定ガイド.md**
- ログイン失敗 / 送信されない / 再送が走らない / bot検知が疑われる → **docs/トラブルシューティング.md**
